# echoim-admin

EchoIM 管理后台前端目录。

当前后台还处于规划阶段，本 README 用于把管理后台在 Telegram 对标路线里的职责先固定下来，方便后续单独立项开发。

## 后台目标

后台不负责复刻 Telegram 的用户端体验，而负责支撑聊天产品逐步走向可治理、可配置、可运营。

## 管理后台优先级清单

### P0

- 用户管理
- 群组管理
- 频道管理基础信息
- 系统设置
- 操作日志

### P1

- 会话申诉 / 删除审核辅助
- 文件资源管理
- 频道 / 群组成员治理
- 黑名单 / 风控策略

### P2

- 邀请链接管理
- Bot 平台管理入口
- 公告、推荐位、系统消息模板
- 数据分析与报表

## 后台任务拆分表

| 阶段 | 模块 | 任务 | 说明 |
|---|---|---|---|
| 第一阶段 | 认证 | 管理员登录 | 独立后台鉴权 |
| 第一阶段 | 用户 | 用户列表 / 状态管理 | 支撑账号治理 |
| 第一阶段 | 群组 | 群组列表 / 基础信息 | 支撑群治理 |
| 第一阶段 | 系统 | 系统配置页 | 与客户端功能开关联动 |
| 第一阶段 | 日志 | 操作日志页 | 支撑审计 |
| 第二阶段 | 文件 | 文件资源查询 | 支撑图片 / 文件消息排障 |
| 第二阶段 | 风控 | 用户 / 群组限制策略 | 支撑治理 |
| 第二阶段 | 频道 | 频道管理页 | 支撑内容型会话 |

## 与客户端路线的关系

当客户端开始推进以下能力时，后台需要同步预留支撑能力：

| 客户端能力 | 后台是否需要支撑 | 说明 |
|---|---|---|
| 真实归档 | 否 | 主要是用户侧状态 |
| 标记未读 | 否 | 用户侧能力 |
| 新建群组 / 频道 | 是 | 后台需要可查看和治理 |
| 图片 / 文件增强 | 是 | 需要文件资源管理 |
| 联系人闭环 | 可选 | 视业务规则决定 |
| 用户名公开体系 | 是 | 需要审核 / 风控支持 |
| Bot 平台 | 是 | 必须有后台治理面板 |

## 建议技术栈

- ECharts

## 当前建议

现阶段不要先做大而全的后台页面。更合理的顺序是：

1. 先补聊天主链路真正依赖的后台能力
2. 再补文件治理和群组治理
3. 最后再做运营分析和平台能力
